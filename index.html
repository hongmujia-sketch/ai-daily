Day 1 | AI 世界的三层结构

2026-06-09
概念纠偏

Claude 不是 Agent，Claude 是模型

很多人（包括从业者）都容易混淆这几个概念。今天最重要的一件事：搞清楚 AI 世界的三层结构。

📱 应用/Agent 层
Claude Code、ChatGPT App、Gemini App
拆解任务、调工具、执行动作
↓
🧠 模型层
Claude、GPT、Gemini、豆包、通义千问
理解语言、思考、生成回答
↓
⚡ 基础设施层
GPU集群、云计算
提供算力
常见误区
"Claude是Agent" → 错。Claude 是模型（大脑），Claude Code 才是 Agent（模型+手脚）。
GPT、Claude、Gemini、豆包全是竞品，都在模型层。

记忆口诀：

  • 模型 = 大脑（只会想和说）
  • Agent = 大脑 + 手脚（能操作电脑、查资料、干活）
  • 应用 = 把 Agent 包装成产品给用户用
基础理论

四个你必须知道的概念

概念类比一句话解释
大语言模型 (LLM) 读了全互联网的学霸 通过海量文本训练出的"语言大脑"
Prompt 给下属的任务简报 你怎么问，决定AI怎么答
幻觉 (Hallucination) 学霸不懂装懂 AI一本正经地胡说八道，因为它在"续写"而非"查证"
Token AI读/写的"字节" AI按token计费，约 1.5个汉字 = 1 token
为什么会有幻觉？
大模型的本质是"续写"——根据前文预测下一个词。它不是在"回忆事实"，而是在"编写最合理的下文"。当它不确定时，宁可编一个听起来对的答案，也不会说"我不知道"。
类比：让一个学霸口试，他不确定的题也会硬答，因为训练目标是"给出流畅回答"而非"承认无知"。
通俗解释：什么叫"预测下一个词"？
想象手机输入法打字，你打"今天天气真"，输入法预测"好"——大模型就是把这个能力放大到亿万亿倍。它每次只做一件事：猜下一个词最可能是什么。所以它不是在"想"，而是在"猜"。

Day 2 | 引擎vs模型 + 三种工作流 + Agent元年真相

2026-06-10
深度理论

「引擎」和「模型」到底什么关系？

在使用 Claude Code 时会看到"推理引擎""模型"这些词交替出现。它们的关系是：

📱 应用层
Claude Code / ChatGPT App
界面和交互体验
↓
⚙️ 推理引擎 (Inference Engine)
接收输入 → 调度模型计算 → 返回结果
↓
🧠 模型 (Model)
一组训练好的参数权重
↓
⚡ 硬件 GPU/TPU 算力集群
类比理解：
模型 = 一本百科全书（静态的知识和能力）
引擎 = 图书管理员（接收你的问题，翻到正确的页面，把答案读给你）
应用 = 图书馆的前台服务（整合了管理员+书，给你完整的体验）

所以当你看到"切换模型"时 = 换了一本书（能力不同）
当你看到"推理引擎优化"时 = 管理员翻书速度变快了（但书没变）
通俗解释：为什么引擎和模型要分开？
想象你点外卖：模型=厨师的菜谱（怎么做菜的知识），引擎=厨房设备（执行做菜的过程）。同一个菜谱（模型）可以用不同厨房（引擎）做——米其林厨房快且贵，家常厨房慢但便宜。所以AI公司拼优化就是在升级厨房，让菜做得更快更省电。

为什么要分开？产品视角的意义：

  • 同一个模型可以被不同引擎加载 → 所以 Claude 模型既能在 Claude.ai 用，也能在 antcc 用
  • 引擎优化 = 不改模型就能让产品更快更便宜 → 这是各家拼成本的关键战场
  • 2026年趋势：推理成本每年下降 10 倍以上，主要靠引擎优化，不是模型变小
深度理论

Prompt 工作流 vs Context 工作流 vs Harness 工作流

在讨论 AI 产品方案时经常会提到这三种工作流，它们是 AI 产品内部"编排任务"的三种方式。理解它们的区别，能让产品运营在技术讨论中不露怯：

类型核心逻辑类比适合什么场景
Prompt 工作流 靠精心设计的"提示词"引导模型按步骤执行 给员工一份详细的 SOP 操作手册 简单、线性的任务。如：写文案、格式转换
Context 工作流 把相关信息（文档、历史对话、数据）注入给模型，让它基于完整上下文做决策 让员工先读完所有背景材料再做判断 需要"理解全局"的任务。如：基于用户画像推荐券
Harness 工作流 一个"调度系统"动态决定：拆成几步、每步用什么模型、怎么验证结果、失败了怎么重试 一个项目经理，根据任务实时分配人手和流程 复杂、多步骤、需要质量控制的任务。如：代码审查、深度研究
核心区别一句话：
Prompt 工作流 = 给AI一张操作单
Context 工作流 = 给AI足够的背景信息
Harness 工作流 = 给AI一个项目经理来调度它

产品经理/运营关心什么？

  • 成本差异大：Prompt 工作流最便宜，Harness 最贵（会调用多个子任务）
  • 质量差异大：Harness 有验证和重试机制，出错率最低
  • Claude Code 就是 Harness 模式：它会拆任务、选工具、验证结果、自动修正
  • 如果做AI推荐功能：大概率用 Context 工作流（注入用户行为数据 → 模型输出推荐结果）

工具进阶

Claude Code 的 Harness 工作流：每天在用的"隐藏功能"

使用 Claude Code 对话时，它背后在做的事远比"回答问题"复杂：

  1. 理解意图 — 判断你要什么（写文件？查数据？讨论？）
  2. 拆解任务 — 如果复杂，自动拆成多步
  3. 选择工具 — 决定是读文件、搜索网页、还是执行代码
  4. 执行 + 验证 — 做完后检查结果是否符合预期
  5. 记忆存储 — 重要信息存到 memory 里下次用

怎么利用这个特性：

  • 给复杂任务而不只是简单问题 — 比如"帮我做一个完整的竞品分析报告"
  • 不用自己拆步骤 — 直接说最终目标，让 AI 来拆
  • 要求"先做计划再执行" — 说"先列个方案让我确认，再动手"

AI 产品的核心指标体系（2026版）

如果你要运营一个 AI 产品（或在产品中加入AI功能），行业现在认的指标是这些：

指标层具体指标对标传统产品为什么不同
效果层任务完成率转化率AI产品的"转化"= 用户交给AI的事是否真正完成了
效果层准确率/幻觉率无AI独有问题——说错了比不说更可怕
体验层响应时长页面加载速度AI生成需要时间，3秒和30秒是两个产品
粘性层重复使用率/周频DAU/次日留存AI产品看"回来再用"比"每天打开"更重要
价值层用户节省时间GMV/收入AI产品的终极价值 = 帮用户省了多少时间/钱
信任层用户修改率无用户改AI输出的比例越低 = 信任度越高
运营启发：如果要做"AI选券"，运营要盯的不只是点击率，而是：
①推荐的券用户实际核销了吗（任务完成率）
②推荐是否准确（有没有推不相关的）
③用户是否信任到直接领取（vs 还要自己翻翻看）

Day 2 检验

1. "推理引擎"和"模型"的关系是？

  • A. 推理引擎是比模型更高级的AI
  • B. 模型是静态的"知识"，引擎是负责运行模型的系统
  • C. 它们是同一个东西的不同叫法
正确答案是 B。模型像一本百科全书（静态参数），推理引擎是图书管理员（接收请求、调度计算、返回结果）。优化引擎=翻书更快，但书没变。

2. 支付有礼要做"AI推荐优惠券"，最可能用哪种工作流？

  • A. Prompt工作流 — 写一个精心设计的提示词就行
  • B. Context工作流 — 把用户画像和行为数据注入给模型做推荐
  • C. Harness工作流 — 需要拆成多步骤并自动验证
正确答案是 B。推荐场景的核心是"基于足够的用户上下文做判断"，需要注入用户数据让模型理解用户偏好。不需要复杂的多步骤编排，一次推理就够了。

3. 2026年AI Agent项目的失败率预计是多少？

  • A. 10% — 技术已经很成熟了
  • B. 20% — 少数企业会失败
  • C. 40%+ — 接近一半会失败，主因是流程没理顺
正确答案是 C。Gartner预测超40%的Agent项目会在2027年前被取消，主因不是技术不行，而是成本失控、价值不清晰、流程没有预先梳理。

4. Google搜索新推出的"资讯代理"会怎样改变用户行为？

  • A. 用户从"主动搜索"变为"等AI主动通知"
  • B. 用户会完全不需要搜索了
  • C. 只是搜索结果的排版变好看了
正确答案是 A。Google的资讯代理可以24h后台跑任务，找到符合条件的结果时主动推送通知。这是"搜索"到"订阅式智能推送"的范式转变，对运营触达策略有根本影响。

Day 3 | AI 是如何"学习"的？——训练、微调、RAG

2026-06-11
核心理论

让AI变聪明的三种方式

AI产品讨论中频繁出现三个概念：训练、微调、RAG。它们是让AI"获得知识"的三种方式，但成本和效果天差地别。

方式类比时间成本适合场景
训练 (Training)从小学到博士毕业数周~数月千万~亿级美元打造一个全新模型
微调 (Fine-tuning)博士后做专业进修数小时~数天数千~数万美元让通用模型变成领域专家
RAG开卷考试（带参考资料）实时极低让AI基于最新资料回答
通俗解释：这三种方式到底有什么区别？
假设AI是一个新员工：
训练 = 让他从幼儿园开始上学直到博士毕业，花了20年。这就是OpenAI做GPT-4时做的事——从零教会模型理解人类语言。
微调 = 这个博士毕业后，公司给他做了一个月的岗前培训，让他专门了解公司的业务。比如让通用的Claude变成一个只懂金融的Claude。
RAG = 不改变他的大脑，但每次问他问题时，递给他一份参考资料让他看着答。这就是为什么AI能回答"公司内部文档里写了什么"——不是它学过，是你临时给了它资料。
产品运营最需要理解的是RAG
因为：训练和微调是技术团队的事（你不需要操心），但RAG直接影响产品设计——"AI能不能回答我们的私域问题"取决于RAG做得好不好。
深入理解

RAG：为什么这是运营最该懂的技术概念

RAG（Retrieval Augmented Generation，检索增强生成）= 让AI在回答问题前，先去检索相关资料，再基于资料生成答案。

用户提问：今天有什么优惠券可以领？
↓
检索系统：去券池/用户数据库里查相关信息
↓
模型：基于检索结果 + 用户画像，生成个性化推荐回答
↓
返回：基于您的消费习惯，推荐这3张券...
RAG 在2026年的行业地位：
已经从"实验性技术"变为"企业AI必选架构"。据行业报告，企业采用RAG后：
- 模型重训成本降低 45%
- AI部署周期加快 2.8倍
- 知识工作者每周节省 11+ 小时
来源：LinkedIn - RAG Strategic Evolution 2026 | 来源：Squirro - State of RAG 2025
通俗解释：为什么RAG比重新训练模型好？
想象你是一个老师，学生问了一个你不知道答案的问题。
重新训练 = 你回去读了10年的书来学这个答案。太贵太慢。
RAG = 你翻开桌上的参考书，找到相关段落，然后用自己的话给学生解释。又快又准。
所以企业做AI产品，几乎都用RAG——因为公司的业务知识在变，不可能每变一次就重新训练模型。
工具实操

实操：你每天都在用RAG，只是不知道

当跟AI对话时加了"参考资料"，就是在用RAG的原理：

  1. 给Claude一段数据再提问 = 手动RAG（你自己做了检索这步）
  2. Claude Code读取本地文件再回答 = 自动RAG（工具帮你检索）
  3. DeepInsight 自然语言查数 = 产品化的RAG（系统自动从数据库检索）
实操技巧：当AI回答不准时，不要只改问法——试试"给它更多参考资料"。
比如不要问"帮我写活动方案"，而是"基于以下去年618的数据和今年的目标，帮我写活动方案：[贴数据]"
你手动提供的参考资料 = 你在帮AI做RAG。

Day 4 | 为什么 Agent 执行任务时很慢？——推理机制与速度真相

2026-06-12
核心理论

AI 的"慢"和 App 的"慢"是两回事

当 Agent 执行一个任务需要 30 秒甚至几分钟时，很多人第一反应是"网络卡了"或"服务器慢了"。实际上，AI 的延迟有完全不同的生成逻辑——理解这个逻辑，是跟技术团队高效协作的前提。

AI 生成回复的三个阶段：

⏱️ 阶段一：TTFT（首Token延迟）
从你发送消息到AI开始打出第一个字的时间
这是"思考准备"阶段
↓
⚡ 阶段二：生成（逐Token输出）
AI在逐字生成回答。速度通常 30-80 tokens/秒
这是"边说边想"阶段
↓
🔄 阶段三：工具调用（Agent专属）
Agent需要读文件、搜网页、执行代码
这是"动手干活"阶段
关键认知：AI 不是"想完了再说"，而是"边想边说"——模型每生成一个词，都需要做一次完整的推理计算。所以回答越长，等待时间越长，这是物理限制，不是"网速问题"。
通俗解释：为什么 AI 不能先想完再说？
想象你在做一道填空题："今天天气真____"。你只能看到一个空，填完这个字后，下一个空才会出现。
AI 也一样——它每"写"一个词，都要基于它刚写的上一个词来决定下一个词。所以必须一个一个来，不能一次性把整篇文章算好再输出。这就像你不能先把一整个对话的每句话都预判好，只能一句一句往下接。
所以回复越长 = 计算次数越多 = 越慢。
深度理论

为什么 Agent 比普通对话更慢？

如果你用过 Claude Code 或 ChatGPT 的深度研究功能，会发现它执行一个任务有时要几分钟。根本原因在于 Agent 的工作流是"多次推理"的：

环节为什么耗时用户感受能不能跳过？
理解任务 需要推理用户意图，拆解为子任务 等待开始 不能
规划步骤 决定"先做什么、再做什么" 看它在思考 简单任务可跳过
执行工具 读文件/查资料/运行代码——这些都是真实操作 等待输出结果 不能，这是Agent的核心
验证结果 检查输出是否正确，出错要重试 感觉重复劳动 可以配置"不需要验证"
输出回复 逐Token生成最终回答 等待完整结果 不能
一句话：Agent 的慢 = 把人类"思考+查资料+检查"的时间压缩到了可见的几秒到几分钟。你之所以觉得慢，是因为习惯了对传统App的"点一下立刻出结果"的期待。

实际对比：

  • 写一封促销文案：纯模型对话 5-10 秒 vs Agent 自行查数据后撰写 30-60 秒
  • 做一份竞品分析报告：纯模型对话不可行（不知道行业数据）vs Agent 搜索后撰写 2-5 分钟
  • 分析上周数据给出优化建议：纯模型做不到 vs Agent 查询数据库后输出 1-3 分钟

实操：理解"响应时间"对运营意味着什么

日常使用中，不同类型的任务会有不同的等待时间——了解这个能帮运营做好用户预期管理：

任务类型预期等待时间用户能接受吗？运营对策
简单问答（"今天有什么活动？"） 1-3 秒 可以 确保 Prompt 简洁；使用更快的模型
内容生成（"写一段促销文案"） 5-15 秒 略慢但可接受 加进度提示"正在为您生成…"

5秒以内：可以做成实时对话（参考ChatGPT体验）
30秒以内：需要加进度反馈，否则用户会认为"崩了"
1分钟以上：建议做成异步通知，用户去做别的事，等AI完成后通知
产品运营视角

推理速度对产品体验的三层影响

如果要在支付运营场景中使用 AI，速度问题直接影响用户留存——而且影响方式是隐性的：

第一层：等待焦虑

传统App的"即时响应"已经培养出用户的秒级期待。当AI推荐券需要3-5秒才能出来时，用户可能已经返回首页了。
类比：把手机App的页面加载从0.5秒变成3秒，DAU会掉 20%——AI的延迟同理。

第二层：信任侵蚀

研究发现，AI的响应时间与用户对其准确性的感知呈U型关系。太快（<1秒）让用户觉得"没认真想"；太慢（>10秒）让用户觉得"卡住了"。理想区间是 2-5 秒。
来源：ACM CHI 2023 - AI Response Timing Study

第三层：成本传导

推理速度直接影响Token消耗。同一个任务用不同推理优化策略，成本可以差 10 倍。在运营场景中，每次AI请求都有实际费用——需要控制每个用户的AI调用成本。

运营要做的事：
① 设定"满意速度"标准——不能让用户因为AI慢而流失
② 设计过渡体验——加载动画、逐步展示结果、预估时间提示
③ 配合技术做"性价比路由"——简单问题用快而便宜的模型，复杂问题才用强的模型

正确答案是 A。AI 的本质是"逐Token生成"——每生成一个词都需要基于已经生成的内容再做一次完整推理。所以回答越长=计算次数越多=越慢，这是物理限制。

2. Agent 比普通对话慢的核心原因是？

  • A. Agent 用的模型更差
  • B. Agent 需要"多次推理+工具调用+验证结果"，是多步骤流程
  • C. Agent 没有经过优化
正确答案是 B。Agent 不是简单的问答，而是要：理解任务→规划步骤→调用工具→验证结果→输出。每一步都是一次或多次独立的模型推理调用，所以总时间自然更长。

  • C. 视任务类型而定——5秒内可实时，超过1分钟建议做异步通知
正确答案是 C。简单问答5秒以内可接受（需进度提示）；30秒以上需明确告知等待时间；超过1分钟建议改为异步模式，完成后通知用户。一刀切的标准不适用于AI产品。
