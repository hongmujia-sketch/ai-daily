Day 1 | AI 世界的三层结构

2026-06-09
概念纠偏

Claude 不是 Agent，Claude 是模型

很多人（包括从业者）都容易混淆这几个概念。今天最重要的一件事：搞清楚 AI 世界的三层结构。

📱 应用/Agent 层
Claude Code、ChatGPT App、Gemini App
拆解任务、调工具、执行动作
↓
🧠 模型层
Claude、GPT、Gemini、豆包、通义千问
理解语言、思考、生成回答
↓
⚡ 基础设施层
GPU集群、云计算
提供算力
常见误区
"Claude是Agent" → 错。Claude 是模型（大脑），Claude Code 才是 Agent（模型+手脚）。
GPT、Claude、Gemini、豆包全是竞品，都在模型层。

记忆口诀：

  • 模型 = 大脑（只会想和说）
  • Agent = 大脑 + 手脚（能操作电脑、查资料、干活）
  • 应用 = 把 Agent 包装成产品给用户用
基础理论

四个你必须知道的概念

概念类比一句话解释
大语言模型 (LLM) 读了全互联网的学霸 通过海量文本训练出的"语言大脑"
Prompt 给下属的任务简报 你怎么问，决定AI怎么答
幻觉 (Hallucination) 学霸不懂装懂 AI一本正经地胡说八道，因为它在"续写"而非"查证"
Token AI读/写的"字节" AI按token计费，约 1.5个汉字 = 1 token
为什么会有幻觉？
大模型的本质是"续写"——根据前文预测下一个词。它不是在"回忆事实"，而是在"编写最合理的下文"。当它不确定时，宁可编一个听起来对的答案，也不会说"我不知道"。
类比：让一个学霸口试，他不确定的题也会硬答，因为训练目标是"给出流畅回答"而非"承认无知"。
通俗解释：什么叫"预测下一个词"？
想象手机输入法打字，你打"今天天气真"，输入法预测"好"——大模型就是把这个能力放大到亿万亿倍。它每次只做一件事：猜下一个词最可能是什么。所以它不是在"想"，而是在"猜"。

Day 2 | 引擎vs模型 + 三种工作流 + Agent元年真相

2026-06-10
深度理论

「引擎」和「模型」到底什么关系？

在使用 Claude Code 时会看到"推理引擎""模型"这些词交替出现。它们的关系是：

📱 应用层
Claude Code / ChatGPT App
界面和交互体验
↓
⚙️ 推理引擎 (Inference Engine)
接收输入 → 调度模型计算 → 返回结果
↓
🧠 模型 (Model)
一组训练好的参数权重
↓
⚡ 硬件 GPU/TPU 算力集群
类比理解：
模型 = 一本百科全书（静态的知识和能力）
引擎 = 图书管理员（接收你的问题，翻到正确的页面，把答案读给你）
应用 = 图书馆的前台服务（整合了管理员+书，给你完整的体验）

所以当你看到"切换模型"时 = 换了一本书（能力不同）
当你看到"推理引擎优化"时 = 管理员翻书速度变快了（但书没变）
通俗解释：为什么引擎和模型要分开？
想象你点外卖：模型=厨师的菜谱（怎么做菜的知识），引擎=厨房设备（执行做菜的过程）。同一个菜谱（模型）可以用不同厨房（引擎）做——米其林厨房快且贵，家常厨房慢但便宜。所以AI公司拼优化就是在升级厨房，让菜做得更快更省电。

为什么要分开？产品视角的意义：

  • 同一个模型可以被不同引擎加载 → 所以 Claude 模型既能在 Claude.ai 用，也能在 antcc 用
  • 引擎优化 = 不改模型就能让产品更快更便宜 → 这是各家拼成本的关键战场
  • 2026年趋势：推理成本每年下降 10 倍以上，主要靠引擎优化，不是模型变小
深度理论

Prompt 工作流 vs Context 工作流 vs Harness 工作流

在讨论 AI 产品方案时经常会提到这三种工作流，它们是 AI 产品内部"编排任务"的三种方式。理解它们的区别，能让产品运营在技术讨论中不露怯：

类型核心逻辑类比适合什么场景
Prompt 工作流 靠精心设计的"提示词"引导模型按步骤执行 给员工一份详细的 SOP 操作手册 简单、线性的任务。如：写文案、格式转换
Context 工作流 把相关信息（文档、历史对话、数据）注入给模型，让它基于完整上下文做决策 让员工先读完所有背景材料再做判断 需要"理解全局"的任务。如：基于用户画像推荐券
Harness 工作流 一个"调度系统"动态决定：拆成几步、每步用什么模型、怎么验证结果、失败了怎么重试 一个项目经理，根据任务实时分配人手和流程 复杂、多步骤、需要质量控制的任务。如：代码审查、深度研究
核心区别一句话：
Prompt 工作流 = 给AI一张操作单
Context 工作流 = 给AI足够的背景信息
Harness 工作流 = 给AI一个项目经理来调度它

产品经理/运营关心什么？

  • 成本差异大：Prompt 工作流最便宜，Harness 最贵（会调用多个子任务）
  • 质量差异大：Harness 有验证和重试机制，出错率最低
  • Claude Code 就是 Harness 模式：它会拆任务、选工具、验证结果、自动修正
  • 如果做AI推荐功能：大概率用 Context 工作流（注入用户行为数据 → 模型输出推荐结果）

工具进阶

Claude Code 的 Harness 工作流：每天在用的"隐藏功能"

使用 Claude Code 对话时，它背后在做的事远比"回答问题"复杂：

  1. 理解意图 — 判断你要什么（写文件？查数据？讨论？）
  2. 拆解任务 — 如果复杂，自动拆成多步
  3. 选择工具 — 决定是读文件、搜索网页、还是执行代码
  4. 执行 + 验证 — 做完后检查结果是否符合预期
  5. 记忆存储 — 重要信息存到 memory 里下次用

怎么利用这个特性：

  • 给复杂任务而不只是简单问题 — 比如"帮我做一个完整的竞品分析报告"
  • 不用自己拆步骤 — 直接说最终目标，让 AI 来拆
  • 要求"先做计划再执行" — 说"先列个方案让我确认，再动手"

Day 3 | AI 是如何"学习"的？——训练、微调、RAG

核心理论

让AI变聪明的三种方式

AI产品讨论中频繁出现三个概念：训练、微调、RAG。它们是让AI"获得知识"的三种方式，但成本和效果天差地别。

方式类比时间成本适合场景
训练 (Training)从小学到博士毕业数周~数月千万~亿级美元打造一个全新模型
微调 (Fine-tuning)博士后做专业进修数小时~数天数千~数万美元让通用模型变成领域专家
RAG开卷考试（带参考资料）实时极低让AI基于最新资料回答
通俗解释：这三种方式到底有什么区别？
假设AI是一个新员工：
训练 = 让他从幼儿园开始上学直到博士毕业，花了20年。这就是OpenAI做GPT-4时做的事——从零教会模型理解人类语言。
微调 = 这个博士毕业后，公司给他做了一个月的岗前培训，让他专门了解公司的业务。比如让通用的Claude变成一个只懂金融的Claude。
RAG = 不改变他的大脑，但每次问他问题时，递给他一份参考资料让他看着答。这就是为什么AI能回答"公司内部文档里写了什么"——不是它学过，是你临时给了它资料。
产品运营最需要理解的是RAG
因为：训练和微调是技术团队的事（你不需要操心），但RAG直接影响产品设计——"AI能不能回答我们的私域问题"取决于RAG做得好不好。
深入理解

RAG：为什么这是运营最该懂的技术概念

RAG（Retrieval Augmented Generation，检索增强生成）= 让AI在回答问题前，先去检索相关资料，再基于资料生成答案。

用户提问：今天有什么优惠券可以领？
↓
检索系统：去券池/用户数据库里查相关信息
↓
模型：基于检索结果 + 用户画像，生成个性化推荐回答
↓
返回：基于您的消费习惯，推荐这3张券...
RAG 在2026年的行业地位：
已经从"实验性技术"变为"企业AI必选架构"。据行业报告，企业采用RAG后：
- 模型重训成本降低 45%
- AI部署周期加快 2.8倍
- 知识工作者每周节省 11+ 小时

通俗解释：为什么RAG比重新训练模型好？
想象你是一个老师，学生问了一个你不知道答案的问题。
重新训练 = 你回去读了10年的书来学这个答案。太贵太慢。
RAG = 你翻开桌上的参考书，找到相关段落，然后用自己的话给学生解释。又快又准。
所以企业做AI产品，几乎都用RAG——因为公司的业务知识在变，不可能每变一次就重新训练模型。

Day 4 | 为什么 Agent 执行任务时很慢？——推理机制与速度真相

2026-06-12
核心理论

AI 的"慢"和 App 的"慢"是两回事

当 Agent 执行一个任务需要 30 秒甚至几分钟时，很多人第一反应是"网络卡了"或"服务器慢了"。实际上，AI 的延迟有完全不同的生成逻辑——理解这个逻辑，是跟技术团队高效协作的前提。

AI 生成回复的三个阶段：

⏱️ 阶段一：TTFT（首Token延迟）
从你发送消息到AI开始打出第一个字的时间
这是"思考准备"阶段
↓
⚡ 阶段二：生成（逐Token输出）
AI在逐字生成回答。速度通常 30-80 tokens/秒
这是"边说边想"阶段
↓
🔄 阶段三：工具调用（Agent专属）
Agent需要读文件、搜网页、执行代码
这是"动手干活"阶段
关键认知：AI 不是"想完了再说"，而是"边想边说"——模型每生成一个词，都需要做一次完整的推理计算。所以回答越长，等待时间越长，这是物理限制，不是"网速问题"。
通俗解释：为什么 AI 不能先想完再说？
想象你在做一道填空题："今天天气真____"。你只能看到一个空，填完这个字后，下一个空才会出现。
AI 也一样——它每"写"一个词，都要基于它刚写的上一个词来决定下一个词。所以必须一个一个来，不能一次性把整篇文章算好再输出。这就像你不能先把一整个对话的每句话都预判好，只能一句一句往下接。
所以回复越长 = 计算次数越多 = 越慢。
深度理论

为什么 Agent 比普通对话更慢？

如果你用过 Claude Code 或 ChatGPT 的深度研究功能，会发现它执行一个任务有时要几分钟。根本原因在于 Agent 的工作流是"多次推理"的：

环节为什么耗时用户感受能不能跳过？
理解任务 需要推理用户意图，拆解为子任务 等待开始 不能
规划步骤 决定"先做什么、再做什么" 看它在思考 简单任务可跳过
执行工具 读文件/查资料/运行代码——这些都是真实操作 等待输出结果 不能，这是Agent的核心
验证结果 检查输出是否正确，出错要重试 感觉重复劳动 可以配置"不需要验证"
输出回复 逐Token生成最终回答 等待完整结果 不能
一句话：Agent 的慢 = 把人类"思考+查资料+检查"的时间压缩到了可见的几秒到几分钟。你之所以觉得慢，是因为习惯了对传统App的"点一下立刻出结果"的期待。

实际对比：

  • 写一封促销文案：纯模型对话 5-10 秒 vs Agent 自行查数据后撰写 30-60 秒
  • 做一份竞品分析报告：纯模型对话不可行（不知道行业数据）vs Agent 搜索后撰写 2-5 分钟
  • 分析上周数据给出优化建议：纯模型做不到 vs Agent 查询数据库后输出 1-3 分钟

推理速度对产品体验的三层影响

如果要在支付运营场景中使用 AI，速度问题直接影响用户留存——而且影响方式是隐性的：

第一层：等待焦虑

传统App的"即时响应"已经培养出用户的秒级期待。当AI推荐券需要3-5秒才能出来时，用户可能已经返回首页了。
类比：把手机App的页面加载从0.5秒变成3秒，DAU会掉 20%——AI的延迟同理。

第二层：信任侵蚀

研究发现，AI的响应时间与用户对其准确性的感知呈U型关系。太快（<1秒）让用户觉得"没认真想"；太慢（>10秒）让用户觉得"卡住了"。理想区间是 2-5 秒。
来源：ACM CHI 2023 - AI Response Timing Study

第三层：成本传导

推理速度直接影响Token消耗。同一个任务用不同推理优化策略，成本可以差 10 倍。在运营场景中，每次AI请求都有实际费用——需要控制每个用户的AI调用成本。

运营要做的事：
① 设定"满意速度"标准——不能让用户因为AI慢而流失
② 设计过渡体验——加载动画、逐步展示结果、预估时间提示
③ 配合技术做"性价比路由"——简单问题用快而便宜的模型，复杂问题才用强的模型

Day 5 | 提示词工程入门——运营必备的 Prompt 技能

2026-06-13
核心理论

提示词工程不是"写指令"，而是"引导推理路径"

很多人觉得写 Prompt 就是"把需求说清楚"，但实际远不止于此。好的 Prompt 不是在给 AI 下命令，而是在帮 AI 建立一条清晰的"思考路径"。理解这个本质差异，你的 Prompt 质量会有质的飞跃。

Prompt 与搜索关键词的本质区别：

维度传统搜索关键词好 Prompt
目标召回匹配内容引导生成路径
方式关键信息匹配上下文+格式+角色+约束
输出已有内容的命中全新内容的生成
代差找到"别人写过的东西"产生"以前不存在的方案"
关键认知：AI 模型本质是一个"接龙游戏"——它预测最可能的下一个词。Prompt 的质量决定了它"从哪个方向开始接"。一个好的 Prompt 不是在"命令"AI，而是在给它布置一个"思考场景"。
通俗解释：为什么 Prompt 这么讲究？
想象你走进一家咖啡店：
❌ 差 Prompt = "给我一个咖啡" → 店员可能给你速溶、拿铁、美式、冷的热的都有可能。
✅ 好 Prompt = "你好，我想要一杯中杯热拿铁，少糖，用燕麦奶，谢谢" → 一步到位。
给 AI 写 Prompt 也一样——上下文越完整、格式越明确、约束越清晰，AI 越不容易"自由发挥"。
深度理论

Prompt 的四大核心要素

一个高质量的 Prompt 通常包含以下四个要素，你可以把它当作一个"检查清单"：

1. 角色设定 (Role)

告诉 AI "你是谁"，这决定了它调用哪个知识库和能力。"你是一个有5年经验的策略运营专家"和"你是一个刚入行的运营助理"——同一个需求，输出质量天差地别。

2. 任务描述 (Task)

清晰说明要做什么。模糊 = 风险。"分析一下数据"和"从用户画像、转化率、流失原因三个维度分析这份支付数据进行归因"——后者让 AI 有明确的分析框架。

3. 输出格式 (Format)

告诉 AI 怎么呈现结果。"给我建议"和"请以表格形式输出，包含：问题描述、原因分析、建议方案、优先级（高/中/低）"——后者直接得到可用的结构化输出。

4. 约束条件 (Constraint)

划定 AI 不要做什么。"基于数据说话"、"不要编造没有统计的数据"、"控制在500字以内"——这是对抗 AI 幻觉第一道防线。

一个实例对比：
❌ 差 Prompt："帮我写一个促销方案"
✅ 好 Prompt："你是一个支付行业的大促运营专家。请为支付宝「支付有礼」618活动设计一个促销方案。要求：① 使用满减形式，门槛不高于50元；② 面向25-35岁女性用户；③ 预算控制在100万以内；④ 以表格形式输出：活动名称、门槛、优惠力度、预期转化率、预算占比。"

看到了吗？好 Prompt 就像在给 AI 画了一张"思维地图"。

Day 6 | AI "乱说话"怎么办？——理解幻觉、评估与信任

2026-06-14
核心理论

幻觉不是 Bug，而是模型的工作方式

很多人第一次用 AI 发现它"编造事实"时会觉得这是 Bug。但实际上，幻觉是生成式 AI 与生俱来的特性——它不是一个"偶尔出错的信息检索系统"，而是一个"永远在编故事的故事家"。

为什么 AI 会"胡说八道"？

根源通俗理解运营最容易碰到的情况
统计预测本性 AI 不"知道"事实，它只预测"最可能的下一个词" 问"上周活动的核销率"——AI 可能编一个数字，因为它是常见的核销率范围
知识截止日期 模型训练数据有截至日期，之后的事它不知道 问"2025年12月的支付数据"——模型训练截止于2024年，它会强行编一个
自信的谎言 AI 没有"不确定"的机制，它总是自信地输出 AI 会说"根据2024年支付宝年报显示……"但其实年报里根本没有这句话
上下文偏移 长对话中，AI 会"忘记"之前说的内容 AI 先说了"核销率5%"，5轮对话后说"核销率提升到15%"——前后矛盾

不同类型幻觉的"危险等级"：

类型例子危险等级运营对策
事实编造编了一个不存在的活动数据🔴 高始终要求AI给出数据来源
数据捏造自己算了一个错误的转化率🔴 高关键数据必须人工复核
张冠李戴把A活动的结果说成B活动的🟡 中明确指定活动名称和日期
过度推理从"DAU涨了"推断出"用户体验变好了"🟡 中要求AI只基于数据说话
无害修饰把"很好"说成"非常好"🟢 低基本上可以接受
关键认知升级：AI 像一个"非常自信的实习生"——它会上知天文下知地理，但它不会说"我不确定"。所以你的工作不是"期待 AI 完全准确"，而是建立一套"检查 AI 输出"的工作流程。这跟在团队里"审核下属的工作"是一样的逻辑。
深度理论

对抗幻觉的五道防线

作为非技术运营，你不需要理解模型内部的数学原理。但这五道防线可以作为你日常使用 AI 时的"安全检查清单"：

第一道：Prompt 约束（最简单、最有效）

在 Prompt 里明确说："如果没有准确数据，请直接说不知道，不要编造"、"所有数据必须基于附件内容，不要自行推断"、"用'可能''推测'等词标注不确定性"。

第二道：提供参考信息

不要把 AI 当"百科全书"——给它参考资料。在对话中粘贴真实的报告、数据表、截图，然后说"仅基于以上内容回答"。这叫 RAG（检索增强生成），是企业级 AI 应用的核心方法。

第三道：要求 AI 自我检查

让 AI 输出后自己检查一遍："请重新审视你的回答，标记出所有你不确定的部分，并说明理由"。研究发现，这个简单步骤可以减少 30-50% 的幻觉。

第四道：交叉验证

同一个问题问两次不同的 AI（比如 Claude 和 ChatGPT），对比回答。如果两者一致，可信度较高；如果矛盾，需要人工查证。

第五道：人工审核（最后的底线）

AI 输出的一切，如果你要用于对外发布或做重要决策，必须有人审核。这不是"不信任 AI"，而是"对结果负责"。

运营的态度应该是：AI 是"初稿生成器"，不是"终稿产出器"。把 AI 当成 "实习生"，你审核它——每次审核都是一次能力提升的机会。

Day 7 | AI 能"记住"我吗？——记忆、上下文与个性化

2026-06-15
核心理论

AI 的"记忆"和人类的记忆完全不一样

当你和 AI 对话时，它看起来"记得"你之前说过的话。但实际上，AI 的"记忆"是一个非常有限且脆弱的机制——理解这个机制，是你设计好的 AI 产品体验的基础。

AI 的三种"记忆"方式：

记忆类型通俗理解容量举例
上下文窗口 (Context Window) "现在聊的这轮对话"——AI 能看到这次会话中的所有内容 Claude 200K tokens ≈ 15万字
GPT 128K tokens ≈ 10万字		 你在一个会话里上传了一篇文章，AI 可以引用文中的内容
会话内记忆 AI 能记住本轮回话中说过的话——但超过上下文窗口后会被"挤掉" 取决于上下文窗口 前5轮你说"我叫小明"，第6轮 AI 还能叫你的名字
跨会话记忆 AI 关掉后再打开——它彻底"失忆"了 默认不保留任何信息 昨天你让 Claude 帮你写了个报告，今天打开——它完全不记得
关键认知：AI 的"记忆"是从零开始的每一次对话。它不"记住"你，而是你每次在对话里"重新告诉它"你是谁。如果要让 AI 记住跨会话的信息，需要人为地保存和注入上下文——这就是"记忆系统"的本质。
通俗解释：AI 的记忆像什么？
想象你有一个"金鱼助理"——每次见面它都忘了你是谁。你只能：
① 每次见面重新自我介绍（手动提供上下文）；
② 或者，把你的信息写在一张卡片上，每次见面给它看这张卡片（记忆系统/Prompt 前缀）。
③ 再高级一点，你的助理有一个笔记本，每次聊完记下来，下次先把笔记本看一遍再和你聊（AI 的记忆数据库）。
这就是所有 AI 记忆系统的本质：一个"笔记本"——你自己写、每次对话前注入。
深度理论

大模型的"上下文窗口"——为什么 AI 会"忘记"最早说的话

上下文窗口是 AI 的一个关键限制。你可以把它理解为一个"正在阅读的桌面"——桌上只能放这么多纸，放不下了，最早的那些就会被推到地上。

上下文窗口的实际影响：

  • 长文档处理：如果你想请 AI 分析一本200页的运营手册，Claude 的 200K 窗口可以装下大部分（约15万字）。但 GPT-4 的 128K 窗口可能只能装三分之二。
  • 长对话退化为"短视"：当对话进行到第50轮时，最开始的设定和约束可能已经被"挤"出了窗口——AI 开始偏离最初的要求。
  • 注意力的"中部迷失"：研究发现，AI 对输入内容的开头和结尾的关注度最高，中间部分容易被忽略。所以关键信息放开头或结尾最有效。

运营如何利用这个特性：

技巧具体做法原理
关键指令放开头在对话最前面放 Prompt 的核心要求利用"开头注意力最高"的特性
定期重申约束长对话中每隔10轮重申一次核心要求防止早期指令被"挤"出上下文窗口
重要信息放结尾最后放的"请基于以上分析输出结论"利用"结尾绝佳性"效应
新开对话代替续聊复杂任务结束后，新开对话重写 Prompt避免上下文窗口被无用信息填满
一句话：上下文窗口就像 AI 的"短期记忆"。它很强大但有上限。你要学会"帮 AI 管理记忆"——重要的事情放开头、关键约束反复说、用完就新开对话。

实操：在 Prompt 中管理 AI "记忆"的方法

你不需要写代码也能管理 AI 的记忆。在日常使用中，这几个技巧立竿见影：

1. "个人资料卡片"法

在每个新对话的第一条消息中，放一段"个人资料"：
"你好，我是支付有礼的策略运营。我们产品的主要用户是25-35岁城市女性，核心指标是核销率和 DAU。我之前已经理解了关于分层推荐和A/B测试的基础概念。今天想讨论的是……"
这样，每次新对话 AI 都不会"失忆"。

2. "关键结论摘要"法

当你完成一段较长的对话后，对 AI 说："请用50字总结这个对话的核心结论和已确认的事项。然后把这作为下次对话的起点。"扔到新对话的开头。

3. "对话分段"法

如果一个任务需要大量轮次，不要在一个对话里做到底。分阶段：
阶段 1 "请帮我分析数据，输出分析结论" → 新对话
阶段 2 "这是我上一轮的分析结论，请基于此输出建议方案" → 新对话

实操建议：你的 AI 使用可以建立"三个文件"——
① 个人背景提示（你是谁、你的产品、你的核心关注点）——每次新对话开头用
② 对话日志模板（每段重要对话结束后总结关键结论）——方便跨对话续接
③ Prompt 模版库（按场景：数据分析、文案、策略建议）——不用每次重写
